Create `shipcard/src/engine/dailyAggregator.ts` — a standalone daily aggregation engine that mirrors the pattern in `aggregator.ts` but groups by UTC date instead of computing all-time totals.

**DailyStats type** (export it):
```typescript
export interface DailyStats {
  date: string;              // ISO date "2026-03-25" (UTC)
  sessions: number;          // unique session count for this day
  messages: number;          // assistant messages (ParsedMessage count)
  userMessages: number;      // user messages for this day (from ParseResult.stats — but see note below)
  thinkingBlocks: number;    // sum of thinkingBlocks across all messages this day
  tokens: {
    input: number;
    output: number;
    cacheCreate: number;
    cacheRead: number;
  };
  costCents: number;         // Math.round(costDollars * 100) — integer cents
  models: Record<string, number>;    // model name -> total tokens used that day
  toolCalls: Record<string, number>; // tool name -> call count that day
  projects: string[];        // unique project names (last path segment) this day
}
```

**Note on userMessages per day:** Since ParseResult.stats.userMessages is a global count (not per-day), and UserEntry items are not in ParsedMessage[], set `userMessages: 0` per day for now. A future enhancement can add per-timestamp user message tracking. This is acceptable because the dashboard can show the global count from SafeStats and the daily chart focuses on assistant activity. Document this limitation with a `// TODO: per-day user message counting requires tracking UserEntry timestamps` comment.

**aggregateDaily function** (export it):
```typescript
export function aggregateDaily(
  messages: ParsedMessage[],
  pricing: { map: PricingMap; version: string }
): DailyStats[]
```

Implementation pattern (mirrors aggregator.ts):
1. Create a `Map<string, DayAccumulator>` where key is `msg.timestamp.slice(0, 10)` (UTC date).
2. DayAccumulator holds: `sessions: Set<string>`, `messages: number`, `thinkingBlocks: number`, `tokens: TokenCounts` (use zeroTokens() pattern), `costRaw: number`, `models: Record<string, number>`, `toolCalls: Record<string, number>`, `projects: Set<string>`.
3. Single pass over messages, identical to aggregator.ts loop structure but bucketed by date.
4. For cost: use `getModelPricing(msg.model, pricing.map)` and `calculateCost(msg.tokens, pricing.pricing)` exactly as aggregator.ts does. Cache pricing lookups per model (same pattern).
5. For project name: copy the `projectNameFromCwd()` helper from aggregator.ts (it's not exported, so duplicate it — it's 3 lines).
6. After the loop, convert Map to sorted DailyStats[] (sort by date ascending).
7. Convert accumulators: `sessions: bucket.sessions.size`, `projects: Array.from(bucket.projects).sort()`, `costCents: Math.round(bucket.costRaw * 100)`.

Imports needed:
- `ParsedMessage` from `../parser/schema.js`
- `calculateCost`, `getModelPricing`, `type PricingMap` from `./cost.js`
- `type TokenCounts` from `../parser/schema.js`

Do NOT import from or modify aggregator.ts. This is a parallel path.
