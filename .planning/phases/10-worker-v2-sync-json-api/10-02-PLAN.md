  <name>Task 1: Create public JSON API routes</name>
  <files>shiplog-worker/src/routes/api.ts</files>
  <action>
Create `shiplog-worker/src/routes/api.ts`:

```
import { Hono } from "hono";
import { cors } from "hono/cors";
import type { AppType } from "../types.js";
import { getUserData, getTimeSeries } from "../kv.js";
```

1. Create `export const apiRoutes = new Hono<AppType>();`

2. **CORS middleware** — apply wildcard CORS to all routes in this sub-app:
   `apiRoutes.use("/*", cors());`
   This handles preflight OPTIONS automatically. Wildcard is appropriate because the data is already public (same as SVG card). Can be tightened in Phase 11 when dashboard origin is known.

3. **GET /:username/api/stats** handler:
   - Extract `username` from path param
   - `const data = await getUserData(c.env.USER_DATA_KV, username)`
   - If null: return `c.json({ error: "User not found" }, 404)`
   - Return `c.json({ data, syncedAt: (data as any).syncedAt ?? null })`
   - Note: `syncedAt` may not exist for v1-synced users — return null gracefully. For v2-synced users, the Worker stores it augmented on the safeStats object (see research Pitfall 3 — option 1: store `{ ...safeStats, syncedAt }` in KV). If 10-01 doesn't augment stats with syncedAt, use the timeseries `generatedAt` as a fallback or return null.

   **Important implementation detail for syncedAt:** The cleanest approach is to check if the stored data has a syncedAt field (v2 sync stores it) and fall back to null (v1 data won't have it). The v2 POST /sync/v2 handler from Plan 10-01 should store `syncedAt` alongside the data. If it doesn't, add it: before `putUserData`, augment the stats: `const enrichedStats = { ...body.safeStats, syncedAt: new Date().toISOString() }` and store that instead. This way GET /api/stats always has syncedAt for v2 users.

   **Coordinate with Plan 10-01:** If 10-01 is already complete and didn't augment syncedAt into stored stats, modify the syncV2.ts handler to store `{ ...body.safeStats, syncedAt }` instead of raw `body.safeStats`. This is a one-line change in putUserData call.

4. **GET /:username/api/timeseries** handler:
   - Extract `username` from path param
   - `const data = await getTimeSeries(c.env.USER_DATA_KV, username)`
   - If null: return `c.json({ error: "User not found" }, 404)`
   - Return `c.json({ data, syncedAt: data.generatedAt })`
   - `generatedAt` is always present in SafeTimeSeries (set by CLI at sync time) — natural syncedAt source.
