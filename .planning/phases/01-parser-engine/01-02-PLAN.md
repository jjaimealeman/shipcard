<objective>
Build the analytics engine that transforms parsed JSONL data into structured stats with cost estimation.

Purpose: This is the brain of ShipLog — it takes raw parsed messages and produces the AnalyticsResult that every interface (CLI, MCP, cards) will render. Cost estimation using LiteLLM pricing is the key differentiator over raw token counts.

Output: Working aggregator that produces summary stats, per-project breakdowns, per-model breakdowns with nested project detail, tool call counts by name, and estimated costs using tiered LiteLLM pricing with a 3-layer cache.
</objective>

    1. Create `src/engine/types.ts` — the AnalyticsResult output shape from RESEARCH.md:
       - `AnalyticsResult` interface:
         - summary: { totalSessions, totalTokens (TokenCounts from parser/schema), totalCost (string, "~$47.20"), modelsUsed (string[]), projectsTouched (string[]), toolCallSummary (Record<string, number>), pricingVersion (string) }
         - byProject: Record<string, ProjectStats>
         - byModel: Record<string, ModelStats>
         - meta: { filesRead, linesSkipped, dateRange?: { since?: string; until?: string } }
       - `ProjectStats`: { sessions, tokens (TokenCounts), cost (string), models (string[]), toolCalls (Record<string, number>) }
       - `ModelStats`: { tokens (TokenCounts), cost (string), byProject: Record<string, { tokens: TokenCounts; cost: string }> }
       - `EngineOptions`: { projectsDir?: string; since?: string; until?: string; json?: boolean }
       - Import TokenCounts from parser/schema.ts

    2. Create `src/engine/cost.ts` — pricing fetch, cache, and calculation:
       - `ModelPricing` type: { input_cost_per_token, output_cost_per_token, input_cost_per_token_above_200k_tokens?, output_cost_per_token_above_200k_tokens?, cache_creation_input_token_cost?, cache_read_input_token_cost?, cache_creation_input_token_cost_above_200k_tokens?, cache_read_input_token_cost_above_200k_tokens? }
       - `PricingMap` type: Map<string, ModelPricing>
       - Constants: LITELLM_URL (the raw GitHub URL), CACHE_TTL_MS (24 * 60 * 60 * 1000), SHIPLOG_DIR (~/.shiplog via os.homedir()), PRICING_CACHE_PATH (~/.shipcard/pricing.json)
       - `buildPricingMap(raw: Record<string, unknown>): PricingMap` — iterate LiteLLM JSON keys, extract only entries that have input_cost_per_token defined. Store as Map for O(1) lookup.
       - `getPricing(): Promise<{ map: PricingMap; version: string }>` — 3-layer cache:
         - Layer 1: Module-level runtime variable. If set, return immediately.
         - Layer 2: Read ~/.shipcard/pricing.json. Check mtime — if less than 24h old, parse and use. Version: "LiteLLM cached YYYY-MM-DD".
         - Layer 3: fetch(LITELLM_URL). On success, write to ~/.shipcard/pricing.json (create ~/.shipcard/ dir if needed with { recursive: true }), set runtime cache. Version: "LiteLLM live".
         - Fallback: If all three fail, use bundled data/pricing-snapshot.json. Version: "LiteLLM snapshot YYYY-MM-DD" (from file).
         - All I/O in try/catch — never crash on pricing failure.
       - `calcTieredCost(tokens, baseRate, tieredRate): number` — from RESEARCH.md. Threshold: 200,000 tokens. Below threshold: tokens * baseRate. Above: (200k * baseRate) + (excess * tieredRate).
       - `calculateCost(tokens: TokenCounts, pricing: ModelPricing): number` — sum of calcTieredCost for all four token types (input, output, cacheCreate, cacheRead) using corresponding pricing fields.
       - `getModelPricing(model: string, map: PricingMap): { pricing: ModelPricing; isDefault: boolean }` — look up model in map. If not found, fall back to "claude-sonnet-4-6" pricing and return isDefault: true.
       - `formatCost(cents: number): string` — returns "~$X.XX" format. The tilde signals estimation.

    3. Create `data/pricing-snapshot.json`:
       - Fetch the current LiteLLM JSON and extract ONLY the Claude model entries (keys containing "claude"). This is the bundled fallback.
       - Include at minimum: claude-opus-4-6, claude-sonnet-4-6, claude-sonnet-4-20250514, claude-opus-4-20250514. Include all pricing fields including cache token costs and above_200k variants.
       - Add a _meta key: { "snapshot_date": "2026-03-25", "source": "LiteLLM model_prices_and_context_window.json" }

    4. Run `npx tsc --noEmit` to verify compilation.

    IMPORTANT:
    - All cost values are in dollars (LiteLLM prices are per-token in dollars, e.g., 0.000003 per token)
    - formatCost should handle very small amounts (< $0.01) and large amounts (> $1000) gracefully
    - Never throw on pricing lookup failure — always fall through to next layer

<task type="auto">
  <name>Task 2: Analytics aggregator — sessions, tokens, tools, projects, models</name>
  <files>
    shipcard/src/engine/aggregator.ts
  </files>
  <action>
    Create `src/engine/aggregator.ts` — the core aggregation logic:

    1. `aggregate(parseResult: ParseResult, pricing: { map: PricingMap; version: string }): AnalyticsResult`
       - Takes the ParseResult from the parser (messages array + sessions map + stats) and pricing data.
       - Single pass over all ParsedMessages to build accumulators:

    2. Accumulator structures (build during single pass):
       - `sessionTokens`: Map<sessionId, TokenCounts> — sum tokens per session
       - `sessionModels`: Map<sessionId, Set<string>> — models seen per session
       - `sessionToolCalls`: Map<sessionId, Record<string, number>> — tool calls per session by name
       - `projectTokens`: Map<projectName, TokenCounts> — sum tokens per project
       - `projectSessions`: Map<projectName, Set<string>> — sessions per project
       - `projectModels`: Map<projectName, Set<string>> — models per project
       - `projectToolCalls`: Map<projectName, Record<string, number>> — tool calls per project by name
       - `modelTokens`: Map<model, TokenCounts> — sum tokens per model
       - `modelProjectTokens`: Map<model, Map<projectName, TokenCounts>> — per-model per-project tokens
       - `totalTokens`: TokenCounts accumulator
       - `totalToolCalls`: Record<string, number> accumulator

    3. For each ParsedMessage:
       - Derive projectName from cwd: `cwd.split('/').filter(Boolean).pop() ?? 'unknown'`
       - Add tokens to all relevant accumulators (session, project, model, model-project, total)
       - Add tool calls (each tool name in message.toolCalls) to relevant accumulators
       - Track sessionId in project sessions set
       - Track model in session/project model sets

    4. After accumulation, compute costs:
       - For each project: sum calculateCost across all its tokens, grouped by model (so tiered pricing applies per-model correctly). Actually — simpler approach: calculate cost per-message during the accumulation pass, accumulate cost-cents alongside tokens. This avoids re-splitting tokens by model after the fact.
       - Better approach: During the single pass, also accumulate `modelProjectCost: Map<model, Map<project, number>>` by calling calculateCost on each message's tokens with that message's model pricing. Then sum up for project totals, model totals, and grand total.
       - For each model, call getModelPricing to get the rate. Cache the lookup to avoid repeated Map lookups.

    5. Build the AnalyticsResult:
       - summary.totalSessions = parseResult.sessions.size
       - summary.totalTokens = totalTokens accumulator
       - summary.totalCost = formatCost(sum of all message costs)
       - summary.modelsUsed = unique models across all messages
       - summary.projectsTouched = unique project names
       - summary.toolCallSummary = totalToolCalls sorted by count descending
       - summary.pricingVersion = pricing.version
       - byProject: for each project, build ProjectStats with sessions count, tokens, cost, models list, toolCalls
       - byModel: for each model, build ModelStats with tokens, cost, and byProject nested breakdown (each project's tokens and cost for that model)
       - meta: { filesRead: parseResult.stats.filesRead, linesSkipped: parseResult.stats.linesSkipped }

    6. Export the aggregate function as the single entry point.

    IMPORTANT:
    - Single pass over messages — do NOT iterate multiple times
    - Cost calculation happens per-message using that message's model pricing — this correctly handles sessions using multiple models
    - Tool call counts: the toolCalls array in ParsedMessage contains one entry per tool_use block, so just count occurrences of each name
    - Sort toolCallSummary by count descending for readability
  </action>
  <verify>
    cd shiplog && npx tsc --noEmit exits with 0.
    aggregator.ts exports aggregate function.
    aggregator.ts imports ParseResult from parser and PricingMap/calculateCost from cost.ts.
  </verify>
